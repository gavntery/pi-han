1 Introduction

[hide]

The ImageSolver script analyzes the geometry of an astronomical image and calculates the transformations necessary for converting pixel coordinates to celestial coordinates. The result is written in keywords of the FITS files so it can be used by other scripts like AnnotateImage or MosaicByCoordinates.

The coordinates are stored in the image following the specifications at fits.gsfc.nasa.gov/fits_wcs.html [1] [2]. This makes them compatible with other applications.

This is not a "blind solver": The script requires to know in advance an approximation to the resolution and coordinates of the center of the image. Usually it is enough that the starting coordinates are inside of the image and that the resolution has an error less that 50%. The algorithm works comparing the image with the position of the stars extracted from a catalog. Then it calculates a transformation matrix that converts from pixel coordinates to celestial coordinates. Since the script is designed for solving optical images, it is initally configured to use the Gnomonic projection that models how most optical systems generate the images. The script allows several other projections for images that cannot be modelled by the Gnomonic projection.

The script can solve distorted images using surface splines and distortion models. The distortion models are compatible with StarAlignment and can be generated using ImageSolver and ManualImageSolver. When a image is solved with the distortion correction activated, the solution is not compatible with the WCS standard. The splines are stored in the properties of the image using a non-standard format. In this case the algorithm also stores in the WCS keywords a solution without the correction that can be used by other applications.

Some images can be difficult to solve because of the noise or low SNR, lack of enough stars or heavy distortions. In these cases ManualImageSolver should be able to solve the image.

2 Usage

[hide]

2.1 Target Images

Figure 1

This section defines the images the script will solve.

Active window

When this option is selected the script will solve the active image in PixInsight. This option is enabled only when there is at least one image open in the application. The solution will be stored as new keywords in the active image but the results are not written to the disk.

List of files

When this option is selected the script will solve a list of files. The results are written to copies of the original images. The names of the output images are composed appending a suffix to the original names.

The list can be managed with the buttons Add files, Remove files and Clear files.

Output file suffix

This suffix is added to the file name of the output files in order to differentiate them from the original files. This option is only available when the option List of files is checked.

2.2 Image Parameters

Figure 2

Only apply optimization

This option requires that the image is already solved. The script doesn't apply the initial alignment algorithm and only tries to optimize the solution.

This option is useful when the image has a rough solution and we need to improve it. For example, after solving the image with ManualImageSolver using very few control points, Only apply optimization allows to get a more precise solution. This option can also be used to test the effect of different values of the parameters of the script.

Coordinates

The Right ascension and Declination fields define the starting point for the algorithm. The coordinates have to be equatorial coordinates using the standard equinox J2000. These coordinates must be inside the image, and it is better that they as near to the center of the image as possible.

If the image has the keywords OBJCTRA and OBJCTDEC the script initializes the coordinates to the values of these keywords. Most capture programs have the capability of reading the position of the telescope and fill these keywords. Since having a good starting position is very important to configure the capture program to fill OBJCTRA and OBJCTDEC if possible.

Search

This button opens a dialog that allows to search the name of astronomical objects in an online search engine (Sesame). The search can be done by name or identifier: "Rossete nebula", "Altair", "M31", "TYC 1058-3399-1". The coordinates of the selected object are copied to the Coordinates fields.

Obs. Date

This field (Observation date) defines the date when the image was taken. It is necessary for calculating the position of the stars using their proper motion.

If the image has a valid DATE-OBS keyword the script initializes this field to the date in this keyword.

Image Scale

The scale of the image can be defined using the fields Resolution or Focal length. These parameters don't need to be exact since the script is able to find its exact value. However the initial value should be as precise as possible since this increases the probability of the image being solved.

The parameter Focal Length can only be used when Pixel size has a value greater than zero. The focal length must be the effective focal length after taking in account focal reducers, Barlows or any effect that modifies the focal length. In some telescopes, like many amateur Schmidt–Cassegrain, the focusing is done moving the main mirror and this modifies the focal length of the system.

The parameter Focal Length is initialized automatically if the image has the keywords FOCALLEN and XPIXSZ.

Pixel size

This parameter defines the size in micrometers of the pixels of the image sensor. This parameter is optional, but if it is not filled, you can not use the focal length for defining the image scale. If you do use this parameter you must set the exact value since the script can not optimize it.

The parameter PixelSize is initialized automatically if the image has the keyword XPIXSZ.

2.3 Model Parameters

Figure 3

The script solves the image comparing it against stars extracted from a star catalog. This section describes how to configure the catalog.

Automatic catalog

When this option is selected the script chooses the best catalog for the estimated field of view of the image.

Local catalog

If this option is selected the script uses a catalog stored locally. In this case you have to configure the path where the catalog is stored.

The script supports two formats:

  • StarGenerator format: The process StarGenerator uses the PPMX astrometric catalog stored in a .bin file. The PPMX catalog can be downloaded from http://pixinsight.com/download/. This catalog has stars of magnitude less than 12.8.
  • Custom catalog format: These files can be created downloading star data from an online catalog. Pressing the download button the script opens the download dialog that allows to select one of the supported catalogs and the working area.
    The catalog downloader has a limit of 2 million of objects. Some catalogs also have a maximum field of view (i.e. the online PPMX catalog has a maximum field of view of 60 degrees). When the catalog has proper motion data the coordinates of the stars are stored for the given epoch.

Online star catalog

VizieR is an online catalog server that provides information of many catalogs. The script allows to use a subset of the available catalogs:

  • BrightStars: This catalog contains the 9100 brightest stars (magnitude<8)
  • TYCHO-2: This catalog is the result of the Hipparcos space telescope and contains the 2.5 million brightest stars (magnitude<13).
  • UCAC3: This catalog contains stars of magnitudes between 8 and 16.
  • PPMXL: This catalog contains stars of magnitude up to 20.
  • Gaia DR1: This catalog contains stars of magnitude up to 20.

You have to select the catalog best suited for your image. For example for very wide field images (focal length<25mm) you should use the catalog BrightStars since it has enough stars and the queries are fast for big fields. PPMXL and UCAC-3 are useful for very deep images since they have very dim stars. However the queries in these two catalogs are very slow for big fields.

VizieR has several mirror servers that provide the same data. You have to select the best server for you, usually the nearest.

Limit magnitude

This parameter limits the number of stars extracted from the catalog to those with magnitude less than this value. This magnitude should be similar to the magnitude of the dimmest stars in the image. If the value is too low the script won't have enough stars to compare, but if it is too high the script will try to match stars of the image to stars that aren't visible. This will slow the process and it could confuse the algorithm and prevent finding a solution or worse: it could find a bad solution.

Automatic magnitude

When this option is selected the script chooses the optimal magnitude limit for the estimated field of view of the image.

2.4 Advanced Parameters

Figure 4

Projection

Most images can be solved using the Gnomonic projection. However some images, specially very wide field images (focal length less than 10-15mm), require other projections. The script provides several projections that could be more suitable for wide angle images.

Align algorithm

The first step for solving an image consists in aligning the image with StarAlignment using the catalog stars as reference. This parameter defines which algorithm StarAlignment uses during the alignment.

There are two options:

  • Triangles: Uses a triangle similarity algorithm which is fast, works in most images but have problems in images with strong distortions.
  • Polygons: Uses an algorithm based on the comparison of polygons which is more tolerant to distortions and scale differences, but it doesn't work with mirrored images.

You should use Polygons when you are sure that the image is not mirrored and Triangles in the other cases.

Star sensitivity

This parameter configures the sensitivity of the star detection. Increase the value to detect less stars.

It is important to set a good value for this parameter so the script detects all the stars of the image and it doesn't mistake noise by stars. The default value of -1 is usually good for lineal images. Stretched images usually require a higher value (0-1).

You can use the option Show stars for checking which stars are being detected.

Show stars

When this option is active the script generates a new image with marks on the position of the detected stars in the original image. It is useful for finding a good value for the Star sensitivity parameter.

Noise reduction

Some images have so much noise that the script could mistake noise by stars. This parameter removes noise from the image in order to improve the star detection.

The value represents the number of wavelet layers that will be removed for noise reduction. Use 0 for no noise reduction.

Generate residuals image

When this option is selected the script generates an image which shows the residuals of the solution. The image shows as green crosses the predicted position of the stars using the calculated solution. Red lines from each star join the predicted position to the actual position of the star.

This image can be used to analyze the errors of the solution. In a good solution the green crosses should be centered on stars and all the red lines should be very short (in a perfect solution the lines should have degenerated to points). When most red lines are long the solution is not good.

2.5 Distortion correction

Figure 6

This parameter activates the distortion correction using surface splines. This option should be selected when the image deviates from an ideal projected image. This usually happens in images taken with focal lengths shorter than 400mm.

When this option is active ImageSolver stores the solution in the properties of the image using a non-standard format. In order to be able to use the solution in other applications, ImageSolver also computes a lineal solution (without distortion correction) and stores it in the keywords of the image in the WCS format.

The options Show distortion map and Generate residuals image can help to evaluate the quality of the distortion correction.

Spline smoothing

Sometimes the algorithm mistakes a star by another star or a noise pixel. When the solution is computed using many stars these errors mostly cancel each other but the final result can be "lumpy". This situation can be detected analyzing the image generated by Show distortion map.

The splines can be smoothed in order to reduce the lumpiness caused by the misidentification of stars. Usually very small values (0 - 0.05) are good enough but sometimes can produce "lumpy" images. Greater values produce smoother images but the correction could not fix distortions too big.

Show distortion map

When this option is activated the script creates a new window with a representation of the distortions in the image. The new image contains a grid with arrows at the center of each cell. The arrows point to the point where the center of the cells would be located if the image didn't have optical aberrations.

This option is only enabled when the distortion correction is active.

Figure 7

Distortion model

When the image has strong geometric distortions a distortion model helps to solve the image. The distortion models used by ImageSolver are compatible with the distortion models of StarAligment. The section Creation of a distortion model explains how to create a distortion model.

Generate distortion model

When this option is active the script generates a distortion model. This model can be used by StarAlignment or by ImageSolver for solving other images.

The model is saved as a text file in the same directory as the image and with the same name and the extension .CSV. This option is only enabled when the distortion correction is activated.

3 What to do when the script doesn't solve an image

[hide]

3.1 General tips

  • Activate the option Show stars and check if the script detects at least 100 stars and that there aren't many false positives. If there are few stars, decrease the value of the Star sensitivity. If there are many false positives increase the value of the Star sensitivity. If there are many noise pixels identified as stars you can increase the value of the noise reduction.
  • The script works better when the starting coordinates or the image scale have a good starting value. If both are far from the real value the script could have problems for finding a solution. For finding a better starting value for the coordinates and the resolution you can search in a planetarium program o celestial chart the coordinates of a star near the center of the image. The coordinates must be in the equinox J2000. For finding an approximation to the resolution of the image is necessary to know the optical system used for taking the image and what geometric transformations have been done to it.
  • The limit of the magnitude of the stars of the catalog has to be matched to the stars visible in the image. You can know the number of stars extracted finding a line in the console like this: 
    Catalog PPMXL size: 154 objects inside the image
    This number should be more than 100 and it should be in the same order of magnitude as the detected stars. Remember to select a catalog suitable for your image:
    • Bright stars: Wide field images with stars up to 8 magnitude.
    • TYCHO-2: Medium field images (50-400mm) with stars up to 12 magnitude.
    • PPMXL: Narrow field images (>400mm).

3.2 Stretched images

The histogram stretch is one of the fundamental steps processing an image. However it can interfere with ImageSolver: when the image is stretched the stars are usually blown up and the noise is magnified. This can make noise to be confused with stars. Also the brightest stars get saturated and sometimes are not identified as stars. This situation can be detected activating the option Show stars.

Fortunately this problem is not very frequent but when it occurs you can solve it "un-stretching" the image:

  • Clone the image
  • Convert the clone to a 32 bits pixel format using SampleFormatConversion.
  • Apply an HistogramTransformation with a midtones value greater that 0.5. This is what un-stretches the image.
  • Activate the option Noise reduction in ImageSolver. Usually values of 1 or 2 are enough.
  • Solve the clone. If ImageSolver is successful copy the coordinates of the clone to the original image using one of the methods in the section How to copy the coordinates between images.

3.3 Wide field images

In this context wide field images are those with a focal length of less than 100mm. These images usually are taken with lenses that have serious geometric distortions that can make quite difficult to solve them. For this kind of images the distortion correction should be always activated.

There are a several strategies for solving these images:

  • Changing the projection: Images taken with very short focal lengths (fish eye lenses, all-sky cameras) can not be modelled with the Gnomonic projection. ImageSolver has projections (i.e. Orthographic, Zenithal Equal Area) that could adapt better to these images.
  • Distortion model: The best method for solving wide field images is using a distortion model. The distortion model allows the script to remove the distortion and simplify the process. The distortion model can be created following the instructions at Creation of a distortion model.
    You can use the distortion model setting its path in the field Distortion model. After a run of the script with a distortion model you can improve the solution running it again activating the option Only apply optimization and deactivating the distortion model.
  • Rough solution with ManualImageSolver: Sometimes the problem is too hard for ImageSolver. In this case you can use ManualImageSolver for getting a rough solution that can be used as a starting point for ImageSolver. A solution with only a few well distributed control points (5-10) is usually enough. The rough solution can be improved executing ImageSolver with the option Only apply optimization. When the image has strong distortions a greater number of control points (20-50) is necessary. It is also very important that the control points cover the entire surface of the image specially the edges and corners.

If neither of these methods works, you can always use ManualImageSolver for solving the image.

4 Creation of a distortion model

[hide]

The distortion models define how the pixels of an image are displaced from its theoretical position if the image were taken by an ideal optical system. The distortion models are usually only necessary for short focal lenses (less than 100-200mm) and most telescopes don't need them. The models have to be calculated for each lens and focal length (for variable focal lenses).

Although creating a distortion model can be quite involved, it only has to be done once for each focal length of each lens. Since most people use only a few lenses for astronomical images this is not usually a problem.

For creating a distortion model you have to follow these steps:

  • First you have to choose a reference image. This image has to be completely covered by stars; it must not touch the horizon or have any part covered by clouds or foreground objects. It is necessary that this image is of a known part of the sky where you are able to identify some of the stars. The best way to get this image is to take a separate shot of a known part of the sky near the zenith. Since the distortion model only depends of the lens and focal length, the resulting model can be used for other images taken with the same optical configuration. If the reference image is partially covered by buildings, trees or the horizon, it can still be used but it won't be able to assist in the distortion correction of those areas.
  • Using ManualImageSolver generate a solution for the reference image. You have to set control points at least at the four corners of the image and the center. 20 well distributed control points should be enough since this solution doesn't need to be very precise. It is necessary only as an starting point for the next step.
    If you select many control points (100-200) you can generate the distortion model directly from ManualImageSolver. In this case you can omit the following steps.
  • Optimize the solution using ImageSolver. This is done checking the option "Only apply optimization". Before optimizing the image activate also the option "Generate distortion model".
  • Check the solution analyzing the residuals images and an annotated version.
  • When you are sure that the solution is the best possible, the distortion model is available at the same directory of the image with the extension .CSV. It is useful to keep the distortion map that ImageSolver generates when the option "Show distortion map". This map can be used for validating the solutions of the rest of the images.

5 How to copy the coordinates between images

[hide]

There are two methods for copying the coordinates from one image to another:

  • FITSHeader.
  • Script CopyCoordinates.

5.1 FITSHeader

The first method can be used when the solution has no distortion correction and both images have exactly the same geometry: same dimensions and the stars in the same exact position. This is usually the case when you extract one of the channels of the image or between the original unprocessed image and the final image if there are not crops nor resizings.

In this case you can follow this instructions:

  • Select in the workspace the image with the coordinates.
  • Open the tool FITSHeader.
  • Activate the Track view mode in FITSHeader. It is the button with the check mark at the right of the lower tool bar.
  • Create an icon with the state of FITSHeader: drag to the workspace the triangle at the left of the lower tool bar.
  • Drag the icon to the target image for the coordinates.

Beware that this copies all the keywords of the source image to the target image.

5.2 CopyCoordinates

The script CopyCoordinates can be used for copying the coordinates of a source image to the image in the active window. This script has two modes:

  • Simple copy: The script copies the coordinates of the source image to the target image without any transformation. This only works when both images have the same geometry so it typically can only be used between cloned images.
  • Transform copy: The script tries to register the target image against the source image. If a transformation is found, the script applies it to the coordinates of the source image and then sets the transformed coordinates on the target image. This can be used when the target image is the result of cropping, scaling or rotating the source image (or any combination of these processes and in general any affine transformation).

The instructions for copying the coordinates from an image to another are these:

  • Select in the workspace the target image for the coordinates.
  • Open the script CopyCoordinates.
  • Select the reference image with known coordinates in the field Source image.
  • Chose between the simple copy and the transform copy. Please read the explanation in the previous paragraphs.
  • Clicking OK the script copies the source coordinates to the target image.

6 Limitations

[hide]

  • ImageSolver is designed for solving deep sky images where enough stars can be detected and matched to a catalog. It can not solve images with too few stars.
  • ImageSolver is not a blind solver which could solve any image knowing nothing about it. You need to know a approximation to the resolution of the image and to the coordinates of the center.
  • ImageSolver can not solve images with discontinuities. This is usually the case of mosaics where the tiles have been joined in a sloppy way.
